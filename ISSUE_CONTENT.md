> - 直接接入蝉妈妈、飞瓜数据等现有平台

### 7. 允灿（服务端开发）2026-03-24 12:24

**我的见解**：收到大家的反馈！我先回应一下被 @的问题，然后补充技术细节：

**回应 @少平 舅舅 - MCP 数据更新频率**：
- MCP 服务本身是"按需调用"模式，没有自动刷新机制
- 我的设计是：后端加一个 **定时任务调度器**（用 Celery + Redis），每小时自动触发一次数据采集
- 采集结果存入 PostgreSQL，Dashboard 读的是数据库，不是实时调 MCP
- 这样即使 MCP 短暂不可用，历史数据依然可查，符合妈妈说的"数据缓存"要求

**回应 @少锋 舅舅 - 错误重试和异常处理**：
- 必须做！我计划三层防护：
  1. **重试机制**：请求失败自动重试 3 次（指数退避：1s → 3s → 9s）
  2. **降级方案**：连续失败 5 次后，标记该账号"数据过期"，Dashboard 显示最后有效数据 + 告警提示
  3. **异常隔离**：单个账号采集失败不影响其他账号，用队列隔离
- 会写详细的错误日志，方便舅舅排查问题

**技术方案细化**：

**数据库设计（初步）**：
```sql
-- 账号基础信息表
CREATE TABLE accounts (
  id SERIAL PRIMARY KEY,
  account_name VARCHAR(100),
  account_id VARCHAR(50),  -- 小红书账号 ID
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 每日指标快照表（时序数据）
CREATE TABLE daily_metrics (
  id SERIAL PRIMARY KEY,
  account_id INTEGER REFERENCES accounts(id),
  metric_date DATE,
  followers INTEGER,
  total_likes INTEGER,
  total_collects INTEGER,
  note_count INTEGER,
  engagement_rate DECIMAL(5,2),  -- 互动率 %
  crawled_at TIMESTAMPTZ DEFAULT NOW()
);

-- 笔记明细表
CREATE TABLE notes (
  id SERIAL PRIMARY KEY,
  account_id INTEGER REFERENCES accounts(id),
  note_id VARCHAR(50),
  title VARCHAR(200),
  publish_date DATE,
  likes INTEGER,
  collects INTEGER,
  comments INTEGER,
  content_hash VARCHAR(64)  -- 内容指纹，去重用

```
